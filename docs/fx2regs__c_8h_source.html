29  This file is a from scratch rewrite of the register
30  definitions for the Cypress FX2 processor. The reason for
31  doing this is that I was unsure of the copyright of all the
32  different versions of fx2regs.h floating around on the net,
33  some where copyrighted by Cypress with no clear license,
34  others were apparently based on the Cypress files but
35  distributed under many varying and sometimes unclear licenses.
36  Most of the files were modified in one way or another by more
37  or less anonymous authors. I wanted an unencumbered header
38  file that I can use for anything, so I decided to rewrite it
39  from scratch and release it under a X11/MIT license.
40 
41  I started with the EZ-USB FX2 Technical Reference Manual (FX2
42  TRM) and wrote a Python script that outputs all the register
43  definitions in this file. I then compared the resulting file
44  to the original fx2regs.h from Cypress and to fx2regs.h from
45  the GNU Universal Software Radio Peripheral (USRP). When the
46  files differed I either changed my definition to match the
47  others or documented why this file differs. Compared to the
48  Cypress header files I've added a lot of symbolic constants.
49  When I've done this I've usually tried to match the
50  definitions from the USRP project if they existed there.
51 
52  Some other differences are:
53 
54  I have skipped a few definitions found in the Cypress header
55  file that are not documented in the FX2 TRM. Some of these
56  seem to be definitions for old FX2 hardware revisions, others
57  seem to be 8051 registers that don't even exist in the EZ-USB
58  family. The registers I've skipped are:
59 
60  APTR1[HL], AUTOPTR1[HL]
61  EXTAUTODAT[12]
62  EP[2-8]GPIFTCH
63  SPC_FNC
64  OUT7BUF
65  CT[1-4], DBUG, TESTCFG, USBTEST
66 
67  I have added definition for AUTODAT1 and AUTODAT2, even though
68  they are not documented in the FX2 TRM. Definitions for these
69  exist in the Keil C header files though and I wanted to add
70  them for completenedd. Do they exist for real on the FX2 or
71  should I remove them?
72 
73  A bunch of invalid sbit definitions from the Cypress header
74  file have been removed (only SFRs with addresses evenly
75  divisible by eight can be bit-addressable).
76 
77  The Cypress header file calls the waveform data
78  GPIF_WAVE_DATA, I've used the name WAVEDATA which matches the
79  FX2 TRM instead. I've skipped the Cypress RES_WAVEDATA_END
80  since I didn't see any use for it.
81 
82  For some reason INT4IN in the Cypress header was called just
83  that, without a bm in front. I've called it bmINT4SRC to
84  match the FX2 TRM and to have a bm prefix as all other
85  bitmasks.
86 
87  The bit called INT6 in the EICON register has been renamed to
88  EINT6 to avoid a name clash with the INT6 bit in the PORTECFG
89  register.
90 
91  I've renamed all I2C bitmask names to start with bmI2C, I did
92  this so that the generic names DONE, ID0 and ID1 would not
93  conflict with all other uses of them in the FX2 TRM. For
94  symmetry I added a bmI2C prefix to the rest of the I2C
95  registers too.
96 
97  The DONE bit in GPIFTRIG is called bmIDLE the same way the
98  USRP project does to avoid a conflict with all other uses of
99  DONE in the FX2 TRM.
100 
101  The DONE bit is GPIFIDLECS is called bmIDLESTATE to avoid a
102  conflict with all other uses of DONE in the FX2 TRM.
103 
104  The defines bmNOAUTOARM and bmSKIPCOMMIT defines from the
105  cypress header files did not match the FX2 TRM. Use the
106  defines bmDYN_OUT and bmENH_PKT that match the FX2 TRM
107  instead.
108 
109  I've added sbit registers called for most bit-adressadble
110  registers that didn't have any before. For example, IOA1 is
111  bit 1 of the IOA register. Is there a better way of doing
112  this? Does the SDCC compiler have a syntax for getting to a
113  bit of a bit-adressable register without having to create a
114  sbit register?
115 */
