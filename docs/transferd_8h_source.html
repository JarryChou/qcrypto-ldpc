23 
24 
25  program to arrange for file transfer from one machine to the other. This is
26  taking care of the sockets between two machines. Each machine has a version
27  of it running. This deamon listens to a command port for file names to be
28  transferred, which are specified relative to a initially agreed directory,
29  and transmitts the underlying file to the other side. On the receiving side,
30  files are saved in a directory, and a notofication is placed in a file.
31 
32  status: worked on more than 55k epochs on feb 7 06
33  tried to fix some errcd packet errors 30.4.06chk
34 
35  usage: transferd -d sourcedirectory -c commandsocket -t targetmachine
36  -D destinationdir -l notificationfile -s sourceIP
37  [-e ec_in_pipe -E ec_out_pipe ]
38  [-k]
39  [-m messagesource -M messagedestintion ]
40  [-p portNumber]
41  [-P targetPortNumber]
42  [-v verbosity]
43  [-b debuglogs]
44 
45  parameters:
46 
47  -d srcdir: source directory for files to be transferred
48  -c commandpipe: where to create a fifo in the file system to
49  listen to files to be transferred. the path has to be
50  absolute.
51  -t target: IP address of target machine
52  -D destdir: destination directory
53  -l notify: if a packet arrives and has been saved, a notification
54  (the file name itself) is sent to the file or pipe named
55  in the parameter of this option
56  -s srcIP: listen to connections on the local ip identified by the
57  paremeter. By default, the system listens on all ip
58  addresses of the machine.
59  -k killoption: If this is activated, a file gets destroyed in
60  the source directory after it has been sent.
61  -m src: message source pipe. this opens a local fifo in the file
62  tree where commands can be tunneled over to the other side.
63  -M dest: if a command message is sent in from the other side, it
64  will be transferred into the file or pipe named in the
65  parameter of this file.
66  -p portnum: port number used for communication. By default, this is
67  port number 4852.
68  -v verbosity: choose verbosity. 0: no normal output to stdout
69  1: connect/disconnect to stdout
70  2: talk about receive/send events
71  3: include file error events
72  -e ec_in_pipe: pipe for receiving packets from errorcorrecting demon
73  -E ec_out_pipe: pipe to send packes to the error correcting deamon
74  -b debuglogs: File to write debuglogs to
75 
76 
77  momentarily, the communication is implemented via tcp/ip packets. the program
78  acts either as a server or a client, depending on the status of the other
79  side. if client mode fails, it goes into server mode. if no connection is
80  available within a few seconds, it tries to connect to the client again.
81 
82  Transfer rationale: The same channel will be used for messages, files and
83  error correction packets. Since there is no simple way to extract the length
84  of the file for all possible future extensions, the transmission of whatever
85  is preceeded by a header involving a
86  typedef struct stream_header {int type;
87  unsigned int length; };
88  were type is 0 for simple files, 1 for messages, 2 for errc packets and
89  length designates the
90  length of the data in bytes. In a later stage, the messages might be sent via
91  an out-of-band marker, but this is currently not implemented.
92 
93 History: first test seems to work 6.9.05chk
94  stable version 16.9. started modifying for errorcorrecting packets
95  modified for closing many open files feb4 06 chk
